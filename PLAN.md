# Plan: Issue #481 — Remove auto-skip gate from SKILL prose

## Summary

The skill prose in `canicode-roundtrip/SKILL.md` (Step 2, lines 72-80) and `canicode-gotchas/SKILL.md` (Step 2, lines 48-53) currently instructs the agent to **skip the gotcha survey entirely** when `isReadyForCodeGen` is `true`. This turns a metadata hint into a hard gate, overriding user choice. The fix removes the gate from both files so the survey is always surfaced (or at least an appropriate prompt to continue is always shown), and `isReadyForCodeGen` becomes purely a banner.

## Current state (exact text to remove)

### `canicode-roundtrip/SKILL.md` — Step 2 (lines 72-80)

```markdown
### Step 2: Gate — check if gotchas are needed

If `isReadyForCodeGen` is `true` (grade S, A+, or A):
- Tell the user: "This design scored **{grade}** — ready for code generation with no gotchas needed."
- Skip directly to **Step 6**.

If `isReadyForCodeGen` is `false` (grade B+ or below):
- Tell the user: "This design scored **{grade}** — running gotcha survey to identify implementation pitfalls."
- Proceed to **Step 3**.
```

### `canicode-gotchas/SKILL.md` — Step 2 (lines 48-53)

```markdown
### Step 2: Check if survey is needed

If `isReadyForCodeGen` is `true` or `questions` is empty:
- Tell the user: "This design scored **{designGrade}** and is ready for code generation — no gotchas to resolve."
- Do NOT write to `.claude/skills/canicode-gotchas/SKILL.md`.
- Stop here.
```

## Three-branch replacement prose

### Branch A: `isReadyForCodeGen=true` AND `questions[]` is empty

No questions were generated by the server — there is nothing to survey. The user is informed and asked to confirm continuing to code generation.

```
Design scored **{grade}** — no gotchas surfaced.

Continue to code generation?
```

In `canicode-roundtrip`, "yes" → proceed to Step 6. In `canicode-gotchas`, "yes" → do not write to SKILL.md; stop here. (Same behaviour as the old skip path, but with explicit user confirmation rather than silent skip.)

### Branch B: `isReadyForCodeGen=true` AND `questions[].length > 0`

The server returned questions even for a high-grade design. Show a choice:

```
Design scored **{grade}** — {N} optional gotcha(s) surfaced.

Would you like to review them, or skip to code generation?
- Reply **review** to go through the questions (recommended if time allows).
- Reply **skip** to proceed directly to code generation.
```

"review" → proceed to survey step (Step 3 in roundtrip, Step 3 in gotchas).  
"skip" → in roundtrip, proceed to Step 6; in gotchas, do not write to SKILL.md; stop.

### Branch C: `isReadyForCodeGen=false`

No change. Present the survey as today.

## Determinism check analysis

`scripts/check-skill-determinism.ts` scans for:

1. **Code-block patterns**: array transforms (`.filter/reduce/flatMap`), `for`/`while` loops, `new Set/Map`, `parseInt/parseFloat`, `.split(`, `.match(/`.
2. **Prose patterns**: `<count of ...>` templates, emoji counting, imperative URL/nodeId/fileKey parsing.

The proposed prose edits contain **none of these patterns**. Specifically:
- The branching is on already-computed fields (`questions` array length, `isReadyForCodeGen` boolean) — both are pre-computed server-side and returned verbatim in the `gotcha-survey` response. This is "branching on already-computed fields", explicitly allowed by ADR-016.
- `questions[].length > 0` is a simple length check on a response field, identical to the existing `questions is empty` check already in `canicode-gotchas` Step 2 (which the checker does not flag today).
- No inline code blocks are added.
- No counting templates or URL parsing is introduced.

The prose check will pass without any new `adr-016-ack` markers.

## Design decisions

1. **Branch on `questions` array, not `isReadyForCodeGen`**: The core issue is that the prose used `isReadyForCodeGen` as a flow gate. `questions[]` is the actual signal — if there are no questions, there is nothing to survey. This split (empty questions = no survey, non-empty + high grade = optional, non-empty + low grade = mandatory) correctly separates the two independent concerns.

2. **User always decides when questions exist**: For Branch B (non-empty + high grade), asking "review or skip?" gives the user control. The old behaviour silently skipped; the new behaviour respects the user's judgment. This directly addresses the real user feedback cited in the issue.

3. **Branch A still results in an explicit user confirmation**: Even when `questions` is empty, the agent asks "Continue to code generation?" rather than silently proceeding. This is a small UX improvement over the old silent skip for the case where the design was perfect.

4. **No deterministic logic added to prose**: The branch conditions (`questions` is empty, `isReadyForCodeGen` is boolean) are pre-computed fields returned by the server. ADR-016 explicitly permits "branching on already-computed fields". The `check-skill-determinism` scanner will not flag these.

5. **Remove the duplicate `questions is empty → skip to Step 6` check in roundtrip Step 3**: Line 96 (`If \`questions\` is empty, skip to **Step 6**.`) becomes redundant because Step 2 now handles the empty case exhaustively. Leaving it would create a confusing double-gate. Remove it.

6. **Wiki Decision Log entry**: Required by acceptance criteria. Entry should record: "hint-vs-gate distinction — `isReadyForCodeGen` is informational, not a flow gate; caught by real user feedback 2026-04-25 (A-grade design auto-skipped survey the user wanted to see)." This is a docs-only note to add to the wiki after the PR is reviewed, not a file in this repo.

## Test strategy

1. **Determinism check passes**: Run `pnpm check:skill-determinism` from the repo root after edits — expect "no ADR-016 violations found".
2. **Manual scenario A** (S/A+ grade, no questions): Run `npx canicode gotcha-survey <high-grade-url> --json`, confirm `questions` is empty, walk through `canicode-gotchas` — agent should say "No gotchas surfaced — nothing to write" and stop after user confirms.
3. **Manual scenario B** (A grade with questions): Simulate with a fixture that has one question; confirm the agent asks "Would you like to review them?" and branches correctly on the answer.
4. **Manual scenario C** (B grade with questions): Confirm no behaviour change — survey is presented as before.
5. **Existing vitest suite**: `pnpm test:run` — no TS source changed, so this should be a no-op verification.

## Risks

- The `canicode-roundtrip/SKILL.md` Step 3 line 96 (`If \`questions\` is empty, skip to **Step 6**.`) must be removed along with the Step 2 edit; if only Step 2 is changed, the duplicate check remains and could confuse future readers.
- The wiki Decision Log entry is part of the acceptance criteria but is not a file in this repo — it must be written after PR review, manually by the author. The plan notes this but cannot automate it.
- Branch B introduces a conversational turn (asking "review or skip?") that wasn't there before. In `canicode-roundtrip`, this adds a round-trip before Step 3, which slightly extends the session. This is the intended trade-off (user control > speed).
